# よくある質問

次に、Adobe Experience Manager Guidesが公開ワークフロー、スケーリング動作、インフラストラクチャのパフォーマンスをどのように管理するかに関する詳細なインサイトを提供する、よくある質問に対する回答の一覧を示します。 Experience Manager Guidesを使用して大規模な公開を行う、エンタープライズユーザー、管理者、および文書作成チームを対象としています。 この図は、Experience Manager Guides Publishing Architectureの全体的なワークフローを示しています。

![](images/IO_runtime.drawio.png)


## Experience Manager Guidesは1日に何件の公開リクエストを実行できますか？

Experience Manager Guidesが1日に処理できる公開リクエストの数は、コンテンツのサイズと種類によって異なります。 設定に従って、システムはプロセッサコアごとに1つの公開ジョブを許可します。 現在の設定では、20個の公開ジョブを並行して実行できます（2個のポッド×各10個のコア）。

実稼動環境が自動的にスケーリングされると、ポッドが最大4にスケーリングされると、この数は40個の同時公開ジョブに増加する可能性があります。

## 公開リクエストをキューに入れる前に、同時に実行できる公開リクエストの数を指定してください。

- 最小（デフォルト）：公開ジョブ 20個（ポッド 2個）
- 最大（拡張）:40件の公開ジョブ（4件のポッド）

この数値は、サーバー全体の負荷によって異なる場合があります。 他のバックグラウンド Sling ジョブが実行されている場合、処理コアが共有され、同時実行数が20未満に減少する可能性があります。

## 複数の公開要求を実行すると、.dita ファイルの編集など、他のアプリケーション機能に影響しますか？

パフォーマンスに影響が出ることはあるかもしれませんが、一般的には最小限です。 多くの処理は、IO ランタイム（サーバーレス公開サービス）で実行されます。一方、AEM インスタンスは、主に依存関係の生成とステータスのポーリングに対してI/O処理を実行します。 その結果、AEM内でのCPUの利用率は低いままであり、オーサリング/編集エクスペリエンスはほとんど影響を受けません。

## Experience Manager Guidesでは、500 KBを超えるSVG や.dita ファイルなどの大きなファイルやグラフィックをどのように管理しますか？

大きなファイルはすべて圧縮され、JCR （Java コンテンツリポジトリ）に保存されます。 IO ランタイムは、公開を開始する前に、完全なzip パッケージを取得します。 複数の大きなファイル（例：各500 KB）を処理する場合でも、最適化されたパッケージと並列I/O処理により、ダウンロードや転送速度に大きな影響はありません。

## Experience Manager Guidesで使用される公開インフラストラクチャと設定は何ですか？

Experience Manager Guidesでは、公開にコンテナ化されたサーバーレスマイクロサービスを使用します。 パブリッシングサービスの新しいバージョンは、Docker イメージとしてリリースされ、Adobe クラウド環境に自動的にデプロイされます。 この設計により、次のことが可能になります。

- お客様向けの専用インフラストラクチャのメンテナンスはありません
- 公開需要を処理するための自動スケーリング
- 迅速な更新と最小限のダウンタイム

## オーバーロードにより公開キューまたは管理システムがクラッシュした場合、他のAEM機能に影響がありますか？

いいえ、Experience Manager Guidesはフォールトトレラントなアーキテクチャで構築されています。 公開キューが過負荷になった場合、リクエストは自動的に再試行され、ポッドは自動的に拡張されて追加の負荷を処理します。 一定のしきい値を超えると、負荷調整が適用されて安定性が維持されます。 その他のアプリケーション領域（オーサリング、レビュー、アセット管理）は影響を受けません。

## Experience Manager Guidesの負荷が大きい場合（Jenkins monitoringと同様）を特定するために、モニタリングツールまたはログアクセスを使用できますか？

いいえ、現在、内部モニタリングツールにアクセスできません。 Adobeの社内チームの場合、次の方法でモニタリングを実行できます。

- ログとエラーの追跡用Splunk
- ポッドレベルのパフォーマンスをチェックし、動作を拡張するためのKubernetes （K8s） CLI

パフォーマンスの低下が確認された場合は、Adobe サポートにお問い合わせください。

## 公開リクエストをマイクロサービスにディスパッチする前にどのような処理が行われますか？

マップコンソールの「プリセット」タブから公開リクエストをトリガーすると、次の手順が実行されます。

1. システムはリクエストを受け入れ、ベースラインと条件付きプリセットを検証します。
2. AEMは、すべての適格なDITA アセットと依存関係を集計します。
3. これらのアセットはzip パッケージにバンドルされています。
4. サーバーレスパブリッシングサービスは、Docker コンテナをスピンアップし、アセットをダウンロードしてパブリッシング処理を実行し、生成された出力アーティファクトをExperience Manager Guidesに戻します。

このワークフローにより、信頼性、分離、スケーラブルな公開の実行が保証されます。

## マップがマイクロサービスコンテナで公開を開始するまでにどのくらいの時間がかかりますか？この時間を定義する要因は何ですか？

公開リクエストは通常、マイクロサービスコンテナにディスパッチされるまでに数分かかります。 この最初の時間は、AEM内の依存関係の集計に使用されます。

リクエストがサーバーレス公開サービスで受信されると、合計公開時間は次の要因によって異なります。

- DITA マップのサイズ
- トピック数とメディアアセット数
- 条件付きコンテンツと書式設定ルールの複雑さ

大きいマップや複雑なマップでは、公開に比例して時間がかかる場合があります。

## Experience Manager Guidesは、キュー内の公開リクエストを（First-Come、First-Serveではなく）優先順位付けできますか？

現在では、すべての公開ジョブは同じように扱われ、先着順（FCFS）モデルに従っています。 現在利用できる優先順位付けメカニズムはありません。

ただし、今後のリリースでは、キューの最適化機能強化の一環として、ジョブのサイズやビジネスの重要度に基づく優先順位付けロジックを導入することができます。

## Experience Manager Guidesでは、公開リクエストの自動スケーリングをどのように処理しますか？

Experience Manager Guidesの公開インフラストラクチャは、読み込みに応じた自動スケーリングをサポートします。 公開需要が増加した場合：

- 追加のポッド（コンテナ）は自動的にスピンアップされます。
- 各ポッドは、複数の公開ジョブを並行して処理できます。
- 負荷が低下すると、ポッドはスケールダウンしてコストとパフォーマンスが最適化されます。

これにより、高可用性、一貫性のあるパフォーマンス、ピーク時のキュー時間の最小化を実現できます。

## 公開ジョブが失敗またはタイムアウトした場合はどうなりますか？

一時的な問題（ネットワークの中断、サービスのタイムアウトなど）が原因で公開リクエストが失敗した場合：

- 公開サービスで設定された再試行ロジックに基づいて、自動的に再試行されます。
- ログとエラーメッセージは、診断目的でバックエンドに取り込まれます。
- 必要に応じて、失敗ステータスを表示し、マップコンソールから手動で公開を再試行できます。

## 公開ジョブの進行状況を監視または追跡できますか？

はい。Experience Manager Guidesでは、マップコンソールの「プリセット」タブで、次のようなリアルタイムのジョブステータスの更新が提供されます。

- ジョブの開始時間と完了時間
- 現在のステージ（結果の送信、ディスパッチ、公開、アップロード）
- エラー通知（存在する場合）

これにより、ジョブの進捗状況を把握し、潜在的な遅延を特定できます。

## Experience Manager Guidesで公開パフォーマンスを向上させるベストプラクティスは何ですか？

最適な公開速度を確保するには、次のベストプラクティスに従ってください。

- 不要な大きな画像ファイルや参照されないトピックを避ける
- ベースラインを使用した公開範囲の制限
- DITA マップ階層を管理しやすく、よく整理しておきます
- オフピーク時間中に多量の公開をスケジュール
- 条件付きフィルターを効果的に使用して、処理負荷を軽減します
