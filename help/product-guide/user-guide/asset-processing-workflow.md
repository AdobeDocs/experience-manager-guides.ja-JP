# アセット処理

アセット処理は、コンテンツアセットがプラットフォーム内で構造化および検証され、インデックス付けされ、アクセス可能であることを確認する重要なワークフローです。 進化するスケーラビリティの要求とクラウドネイティブの要件により、アーキテクチャは、単一スレッドの階層処理モデルから、分散したグラフ対応のマルチスレッドシステムへと大きく変化しました。

## 現在のアセット処理ワークフロー

### 処理の概要

アセットがExperience manager Guidesに読み込まれると、次の連続処理手順が実行されます。

- 一意のキーの割り当て：各ドキュメントに一意のIDを割り当てることで、トレーサビリティと参照の整合性を確保します。
- 解析：コンテンツ（例：DITA XML）は、システムレベルの理解のために構造化コンポーネントに解析されます。
- 検証：構造化およびスキーマの検証により、ドキュメント標準への準拠が保証されます。
- 参照解決：相互参照（リンク、画像、依存関係）は、アセット間で解決されます。
- メタデータの抽出：タイトル、作成者、カスタム属性などのメタデータを抽出して、インデックス作成と検索用に使用します。
- 変更時の再処理：増分再処理は、コンテンツの更新後に一貫性を確保します。

### 建築の特徴

- **単一スレッド処理**: B ツリー実装に依存するJCR （Java コンテンツリポジトリ）構造の破損を防止します。これにより、データの整合性は確保されますが、一括取り込み時に処理のボトルネックが発生します。

- **親マップ依存関係**: グラフ トラバーサルを使用して、アセット間の階層関係を維持します。 これは、計算上のオーバーヘッドが増加し、トラバーサルが多い操作による歪みがある大規模処理中に高い遅延を伴う集中的な操作です。

## 新しいアセット処理フロー

コア処理ステップは機能的に一貫性を保ちますが、現在は分散および並列化されたフレームワーク内で実行され、スループットが大幅に向上します。

### アーキテクチャの強化

- **グラフデータベース統合**:
   - 階層JCRからネイティブグラフデータベースへの移行
   - 関係と依存関係を効率的に処理できる
   - 階層的なストレージでのグラフ操作のシミュレーションの複雑さを排除
- **マルチスレッド分散処理**:
   - 処理は、クラウド環境の複数のポッドにわたって実行されます
   - 単一の引出線ノードへの依存を削除します
   - 水平スケーラビリティと並列実行が可能
- **親マップの依存関係の削除：**
   - 明示的なグラフトラバーサルの必要性を排除
   - I/O操作と処理待ち時間を短縮
   - 処理パイプラインを簡略化
- **同期された一意のID割り当て**
   - 一元化された調整により、次のことが可能になります。
   - ドキュメント IDの重複がありません
   - 分散ノード間の一貫性
   - 同時環境で参照整合性を維持する
- **クラウドネイティブ スケーラブル データベース （AWS ホスト）**
   - 可用性と回復力に優れたデータベースレイヤー
   - ワークロードに応じて柔軟に拡張
   - システム全体の信頼性とパフォーマンスが向上

## 新しいアーキテクチャの利点

- パフォーマンスの向上：
   - 並列実行により、処理時間が大幅に短縮されます
   - トラバーサル作業を排除して待ち時間を短縮
   - グラフ処理の最適化により、依存関係の解決スピードが向上
- 拡張性：
   - ポッド間の水平スケーリングにより、大きな取り込みボリュームを処理できます
   - クラウドネイティブなインフラストラクチャにより、ワークロードの負荷に動的に対応
- 信頼性と可用性：
   - 分散処理により、単一障害点を排除
   - AWSでホストされるデータベースは、高い可用性と耐障害性を保証します
- 効率性の向上：
   - 親マップのトラバーサルの削除によるI/O オーバーヘッドの削減
   - コンピューティングノード全体のリソース稼働率の向上
- データの整合性：
   - 同期IDの割り当てにより、分散システム間の一貫性を維持
   - 同時実行を可能にしながら堅牢性を維持

## データベースの設定

Experience Manager Guidesを使用すると、AEM Cloud環境のデータベース設定を合理化できます。 AEM Cloud インスタンスのデータベースを設定するには、次の手順を実行します。

1. AEM Cloud Managerにアクセスする：以下のURLを使用してAdobe Experience Cloud Managerに移動し、プレースホルダーを組織、プログラム、環境の詳細に置き換えます：`https://experience.adobe.com/#/${orgName}/cloud-manager/environments.html/program/${programId}/environment/${envId}`

1. 環境の設定：Cloud Managerを使用して環境設定ページを開くと、必要なデータベース設定の設定など、インスタンスに固有の設定を調整できるようになります。

この合理化されたアプローチにより、Adobeクラウドインフラストラクチャ内のAEM環境に簡単にアクセスして設定できます。

1. 以下のプロパティを設定します。


| プロパティ名 | 値 | 適用されるサービス | タイプ |
|----------------------------------|--------------------------------|-----------------|----------|
| DATABASE_URL | `<host>:<port>/<db_name>` | 作成者 | 変数 |
| GUIDES_ENABLE_DATABASE | `true` | 作成者 | 変数 |
| DATABASE_PASSWORD | `password` | 作成者 | シークレット |
| DATABASE_USERNAME | `username` | 作成者 | 変数 |
| RUN_POSTPROCESS_IN_PHASES | `true` | 作成者 | 変数 |
| DATABASE_CONNECTION_POOL_SIZE | `10` | 作成者 | 変数 |


![](images/environment-config.png){width="350"}

1. 変更を保存：設定を変更した後、Cloud Manager インターフェイス内で&#x200B;**保存**&#x200B;してください。

1. システムの可用性：設定が完全に適用されたら、GET `http://host/bin/guides/v1/system/status`を開き、以下のプロパティを確認します。
   - `<isDatabase>`: trueである必要があります
   - `<databaseConnectionCheck>`：を渡す必要があります

   上記の値が応答で正しい場合は、新しく設定されたデータベースでシステムを使用できます。

このプロセスに従うことで、AEMを適切に設定し、すぐに使用できます。

>[!NOTE]
>
> 既存のコンテンツを含む既存の環境に移行する場合は、新しいコンテンツを取り込む前に、まずフェーズ 2 （既存のコンテンツを移行）を実行する必要があります。 これは、新しいコンテンツに一時的なGUIDが正しく割り当てられていることを確認するためです。

## JCRからデータベースへのコンテンツの移行（フェーズ 2）

Experience Manager Guidesをデータベースベースの設定に切り替える前にアップロードおよび処理されたコンテンツを移行するには、移行スクリプトを使用する必要があります。 このスクリプトは、移行プロセスを開始および監視するために一連のAPIを活用します。 推奨されるアプローチの概要を以下に示します。

1. 任意のREST クライアントを使用して移行APIをトリガーします。
2. 移行の進行状況を確認します。
3. 完了するまで移行を監視する：進捗状況APIが100%の完了を報告するまで監視を続けます。 完了すると、JCR リポジトリから以前にアップロードおよび処理されたすべてのコンテンツがデータベースに移行されます。

   >[!NOTE]
   >
   > - AEMでリクエストを認証するために、必要な認証ヘッダー（OAuth トークン、API キー、開発者コンソールからのアクセストークンなど）が含まれていることを確認します。
   > - 移行期間は、コンテンツリポジトリのサイズによって異なります。 定期的な進捗チェックと、エラーや中断のモニタリングが推奨されます。
   > - 移行ログがある場合は、移行のパフォーマンスを評価し、問題を特定します。

4. 大きなリポジトリサイズの移行をサポートするには、次の設定に従います

   >[!NOTE]
   >
   > この設定は、移行中にリポジトリトラバーサルエラーが発生した場合にのみ適用します。

   `file name: `org.apache.jackrabbit.oak.query.QueryEngineSettingsService.xml&quot;

   ```xml
   <?xml version="1.0" encoding="UTF-8"?>
   <jcr:root xmlns:jcr="http://www.jcp.org/jcr/1.0"
         xmlns:sling="http://sling.apache.org/jcr/sling/1.0"
         jcr:primaryType="sling:OsgiConfig"
         queryLimitInMemory="5000000"
         queryLimitReads="1000000"
   />
   ```


## 移行API

### 移行を開始

- エンドポイント：`POST /bin/guides/v1/assets/process`
- リクエスト本文：（`application/json`）:

```json
  {
    "path": "/content/dam/dita-templates",
    "excludedPaths": [
      "/content/dam/demo",
      "/content/dam/demo1"
    ],
    "type": "ASSET_PROCESSING"
  }
```

- 移行のprocessingIdを返します。
- トリガー：製品に組み込まれているアセット処理ワークフローです。

### 移行ステータスの確認

エンドポイント：`GET /bin/guides/v1/assets/process/status?processingId=<processingId>`

### 実行中の移行をキャンセル

- エンドポイント：`POST /bin/guides/v1/assets/process/cancel`
- リクエスト本文（アプリケーション/json）:

  ```
  {
   "processingId": "processingId"
  }
  ```

### 失敗またはキャンセルされた移行の再開

- エンドポイント：`POST /bin/guides/v1/assets/process/resume`
- リクエスト本文（アプリケーション/json）:

  ```
  {
   "processingId": "processingId"
  }
  ```

## AEM cloudへのアセットのアップロード

Adobe Experience Manager（AEM）as a Cloud Serviceには、コンテンツの一括取り込みに対する複数のアプローチが用意されています。 特にExperience Manager Guidesの後処理において最適なパフォーマンスを確保するには、サポートされたスケーラブルな取り込み戦略を採用することが不可欠です。

### クラウドストレージ統合を使用した一括取り込み

AEMは、サポートされているクラウドストレージプロバイダーを通じたコンテンツの一括取り込みをサポートしています。これにより、企業は任意のストレージソリューションを接続し、コンテンツをAEMに直接インポートできます。 このアプローチは、次の利点により、大規模な取り込みやパフォーマンスに影響を受けやすい取り込みに適しています。

- スケーラブルなインフラストラクチャ：取り込みプロセスは、Adobeで管理されるクラウドインフラストラクチャ上で実行され、負荷とコンテンツ量に基づく自動スケーリングが可能になります。 これにより、大規模なデータセットであっても、取り込みパフォーマンスの一貫性を維持できます。

- 予測可能な取り込み計画：ユーザーは取り込み時間を事前に推定できるため、リリースの計画、スケジューリング、依存チームとの調整に役立ちます。

  ![](images/ingestion-time.png){width="350"}

- 包括的なログとトラッキング：取り込みワークフローには、詳細なログとトラッキング機能が含まれており、インポートプロセス全体の進行状況、成功状態、潜在的な問題を可視化できます。

  ![](images/bulk-import-job.png){width="350"}

- スケジュール済みの取り込み：コンテンツの取り込みは、事前に定義された時間枠の間にスケジュール可能で、エンドユーザーや継続的な運用に対する影響を最小限に抑えることができます。

詳細については、「[一括読み込みを使用](https://experienceleague.adobe.com/ja/docs/experience-manager-learn/cloud-service/migration/bulk-import)」を参照してください。

## AEM アップロードを使用した一括取り込み

AEMには、ローカルファイルシステムから直接コンテンツを取り込むことができるライブラリおよびコマンドラインインターフェイス（CLI）である[AEM upload](https://github.com/adobe/aem-uploa)も用意されています。 このオプションは、カスタムソリューションに統合するか、スタンドアロンのCLI ツールとして使用してコンテンツをアップロードできます。

このアプローチはユーザーのローカルマシン上で実行されるため、信頼性の高いシームレスな取り込みエクスペリエンスを実現するには、安定した中断のないネットワーク接続が必要です。


## Experience Manager Guides データベース接続のヘルスチェック

データベースを使用するように設定されたExperience Manager Guidesのデプロイメントでは、信頼性を維持するために、安定した一貫性のあるデータベース接続が必要です。 データベース接続ステータスを検証することは、システム機能に影響を与える可能性のある接続関連の問題を除外するための重要なヘルスチェック手順です。

このヘルスチェックを使用すると、データベースが設定され、到達可能であり、期待どおりに機能しているかどうかを確認できます。 DB接続ステータスを確認するには、次の手順に従います。

1. ブラウザーまたはREST クライアントを開く
2. この[URL](https://host:port/bin/guides/v1/system/status)を使用してGET呼び出しをトリガーします
3. 以下のフィールドを使用して、システム設定とヘルスステータスを決定できます
   1. isDatabase:
      - true：環境がデータベースで設定されています。
      - false：環境がデータベースを使用していません

   2. databaseConnectionCheck:
      - 合格：Experience Manager Guidesは接続ステータスを確認し、Guidesがデータベースに接続できる場合は、渡されたステータスを返します。
      - 失敗：環境がデータベースと通信できません。 お客様は、システムの使用を直ちに停止し、Adobe サポートに連絡する必要があります。

## ログの監視

Experience Manager Guides with Databaseは、詳細を効率的にログに記録し、insightにシステムステートを与えます。
Splunkで以下のクエリを使用して、様々なシナリオのログを取得します。

1. 移行ログ：
   - `index IN ("dx_aem_engineering") aem_service=cm-${programid}-${environmentId} sourcetype=aemerror "AssetProcessingJob" OR "AssetJobProducerDb" NOT "ServiceEvent"`
2. 後処理ログ：
   - `index IN ("dx_aem_engineering") aem_service= cm-${programid}-${environmentId} sourcetype=aemerror com.adobe.fmdita.uuid.concrete.Cor*`


>[!NOTE]
>
> [Splunk クエリ機能](https://www.splunk.com/en_us/blog/learn/splunk-cheat-sheet-query-spl-regex-commands.html)の詳細を参照して、時間、ログレベル、または特定のパターンの検索に基づいてこれらのログをフィルタリングします。
